Enforcement

Consistency enforced — not requested

Controls that prevent overrides, regressions, and one-off styling — so releases stay predictable.

1. Strict naming

Token paths are fixed. No ad-hoc names or synonyms.

Prevents token sprawl and lookup errors.

Allowed: color.text.primary
Blocked: darkText, textMain
Impact: fewer orphan tokens → faster fixes

2. Token access rules

Components can only use semantic tokens — primitives are locked.

Removes inline overrides and design drift in code.

Primitive: #111111
Semantic: color.action.primary.surface
Impact: fewer runtime mismatches → lower QA load

3. Component assembly

Components are composed from tokens — no custom values inside components.

Finite variants only — no one-offs.

Button = size: space.control.md
Button = padding: space.control.sm
Impact: fewer hotfixes after releases

4. Mode & scale

Modes and density propagate from tokens so light/dark and accessibility adjustments are automatic.

Prevents inconsistent mode behavior across screens.

Token: text.primary
Light → #111111 · Dark → #FFFFFF
Impact: fewer visual regressions when toggling modes

5. Change control

Semantic changes require a review and are versioned; breaking changes are visible before release.

Allows safe evolution without silent regressions.

Allowed: reviewed semantic change
Blocked: silent primitive swaps / component overrides
Impact: auditable changes → predictable releases
System anatomy

How the system prevents drift

Practical layers that turn scattered UI values into enforceable rules — reducing rework and bugs.

Primitives

Raw values: colors, spacing, typography
Example: #0A0A0A · 16px · Montserrat

Semantics

Roles built from primitives: text.primary, surface.action
Example: color.text.primary = #0A0A0A

Components

Assembled from semantic tokens, enforceable in code
Example: Button uses color.text.primary + space.control.md
